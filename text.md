## [00:00–03:52] 규칙 비대화의 함정과 컨텍스트 노이즈

클로드 코드(Claude Code)를 현업에서 제대로 활용하기 위해 반드시 알아야 할 메모리 관리 고급 스킬을 다룬다. 특히 대규모 프로젝트에서 클로드에게 `CLAUDE.md`라는 업무 매뉴얼을 쥐여주었음에도 AI가 이를 무시하고 엉뚱한 코드를 짜는 현상의 원인과 근본적인 해결책을 제시한다. (발표자는 영상 초반, 자신의 목소리가 AI 목소리라는 오해에 대해 실제 마이크로 직접 녹음하고 있음을 해명하며 시작한다.)

---

### 💡 핵심 내용
프로젝트 규모가 커질수록 단일 메모리 파일에 규칙이 누적되면서 '컨텍스트 노이즈(Context Noise)'가 발생하며, 이를 방지하려면 필요한 규칙만 필요한 시점에 불러오는 '온디맨드(On-demand)' 로딩 설계가 필요하다.

**📌 근거와 논리**

1. **카페 바리스타 비유** — 바리스타에게 가장 중요한 라떼 레시피를 가르치는 와중에 청소하는 법, 재고 정리하는 법, 마감하는 법까지 한꺼번에 쏟아내면 어떻게 될까? 결국 정작 중요한 라떼 레시피는 수많은 부가 정보들 속에 묻혀 잊히고 만다. 클로드 역시 한 번에 너무 많은 규칙을 주면 지시사항을 누락한다.
2. **프로젝트 성장 단계별 규칙의 한계** —
   - **초기 (50줄 이하)**: 실행 명령, 이름 짓는 규칙만 있어 잘 동작함.
   - **성장기 (150줄 내외)**: API 규칙, 테스트 규칙이 추가되어도 아직은 처리 가능.
   - **대규모 (300줄 이상)**: 서버, 팀별 규칙, 버전 이력이 뒤섞이면서 AI가 혼란을 겪고 규칙을 무시하는 문제가 본격적으로 시작됨.
3. **Dev 커뮤니티 개발자 An Vo의 모노레포 실패 및 개선 사례** — 프론트엔드와 백엔드 핵심 로직이 합쳐진 모노레포 환경에서 `CLAUDE.md` 파일이 47,000단어까지 불어나는 문제가 발생했다. 그 결과 클로드의 반응 속도가 심각하게 느려지고 지침을 이행하지 못했다. 이를 해결하기 위해 규칙을 조건부로 나누어 호출하도록 아키텍처를 변경했다.

```text
[Before vs After: 규칙 분리로 인한 로딩 텍스트 감소]

Before:
- 루트 CLAUDE.md: 47,000자 (프론트엔드, 백엔드, 핵심 로직 규칙이 모두 섞여있음)

After (필요한 시점에만 분할 로드):
- 루트 CLAUDE.md: 8,982자 (약 80% 감소)
- frontend/CLAUDE.md: 8,647자
- backend/CLAUDE.md: 7,892자
- core/CLAUDE.md: 7,277자
```

→ 프로젝트 규모가 커짐에 따라 지침이 단일 파일에 무분별하게 누적된다 → 이는 AI에게 정보 과부하를 일으켜 정작 현재 작업에 필요한 핵심 규칙을 망각하게 만든다 → 따라서 파일을 단순히 쪼개는 것을 넘어, '작업 시점'에 맞게 규칙이 자동 호출되도록 설계해야 한다

**🌐 맥락**
`CLAUDE.md`의 핵심 원칙은 "구체적으로 작성하되, 간략히 작성하는 것"이다. 그러나 실무에서 프로젝트가 커지면 API, 테스트, CSS, DB 등 수많은 도메인의 규칙이 필연적으로 추가된다. 개발자들은 성실하게 매뉴얼을 업데이트하지만, 컨텍스트 윈도우(Context Window)가 혼탁해지면서 AI의 퍼포먼스가 급락하는 역설적인 상황에 직면하게 되었다.

**➡️ 시사점**
규칙 파일을 관리하는 전략은 단순한 '파일 분리'가 아니라 '컨텍스트 통제'에 있다. 클로드가 처음에 읽어야 하는 루트 메모리의 용량을 최소 80% 이상 덜어내고, 현재 작업 중인 도메인의 규칙만 머릿속에 들어가도록 환경을 구성하는 것이 대규모 AI 개발 워크플로우의 첫걸음이다.

> "규칙이 계속 쌓이다 보면 지금 필요한 규칙이 관련 없는 규칙들 사이에 묻혀 버리는 거예요. 핵심은 단순히 파일을 쪼갠 게 아니라 필요한 시점에만 불러오게 설계한 거죠." [[02:22](http://www.youtube.com/watch?v=FBv8hK_DtJ8&t=142)]

## [03:52–07:55] 전략 1: 조건부 자동 로딩과 오픈소스 실사례

클로드 코드의 지침을 쪼개는 방법은 크게 3가지(1. `.claude/rules/` 디렉터리와 프론트매터 활용, 2. 하위 디렉터리별 `CLAUDE.md` 배치, 3. 상위 디렉터리 `CLAUDE.md` 적용)가 존재한다. 본 섹션에서는 그중 가장 유용하게 쓰이는 첫 번째 전략인 '프론트매터(Frontmatter)를 이용한 특정 파일 조건부 로딩'의 원리와 실제 오픈소스 적용 사례를 심층 분석한다.

---

### 💡 핵심 내용
`.claude/rules/` 디렉터리 하위에 주제별 마크다운 파일을 생성하고, 파일 최상단에 프론트매터(Frontmatter)로 조건을 걸어두면 클로드가 특정 확장자나 폴더를 수정할 때만 해당 규칙을 온디맨드로 자동 로드한다.

**📌 근거와 논리**

1. **맥도날드 매뉴얼 비유** — 맥도날드 매장에는 주방 직원용, 카운터 직원용, 배달 직원용 매뉴얼이 각각 분리되어 있다. 주방 직원이 배달 직원의 매뉴얼을 읽을 필요가 없듯, 클로드 역시 테스트 코드를 작성할 때 API 라우팅 규칙을 알 필요가 없다.
2. **프론트매터(Frontmatter) 구문 규칙** — 메모리 파일 상단에 대시 3개(`---`)로 감싼 영역을 통해 규칙이 적용될 패턴(paths 또는 globs)을 명시할 수 있다.

```yaml
---
globs: src/api/**/*.ts, src/routes/**/*.ts
---
# API 개발 규칙
- 모든 화면에서 입력값 확인 필수
- 에러가 나면 정해진 형식으로 응답
```
*(위와 같이 설정하면 `src/api/` 및 `src/routes/` 하위의 타입스크립트 파일을 건드릴 때만 이 API 규칙이 활성화된다.)*

3. **실사례 1: trigger.dev (14K Stars)** — 반복 작업을 자동으로 실행해 주는 플랫폼인 trigger.dev는 `.claude/rules/` 하위에 5개의 주제별 메모리 파일(`database-safety.md`, `docs-writing.md`, `legacy-v3-code.md`, `sdk-packages.md`, `server-apps.md`)을 운용한다. 특히 `database-safety.md`에는 "데이터베이스 삭제는 반드시 승인을 받고 진행할 것"이라는 엄격한 규칙이 들어있으며, DB 관련 파일을 수정할 때만 이 규칙이 로드되도록 프론트매터가 적용되어 있다.
4. **실사례 2: CockroachDB (32K Stars)** — 대용량 분산 데이터베이스인 CockroachDB는 모든 Go 언어 파일(`.go`)에 개인정보 보호 처리 규칙을 자동으로 적용한다. 에러 메시지나 로그를 출력할 때 개인정보(PII)가 될 수 있는 항목은 반드시 마스킹 처리 가려진 상태에서 출력하도록 강제한다.

→ 특정 파일에 대한 작업 지침을 전역(`CLAUDE.md`)에 선언하면 모든 작업에 방해를 준다 → 이를 주제별로 분리하고 프론트매터에 적용 경로(`paths/globs`)를 선언한다 → 클로드는 사용자가 지정한 특정 파일(예: DB나 Go 언어 파일)을 건드릴 때만 해당 규칙을 불러와 개인정보 유출이나 DB 날림 같은 치명적 실수를 원천 차단한다

**🌐 맥락**
글로벌 탑티어 수준의 오픈소스 프로젝트들은 이미 AI가 코드를 작성하거나 수정할 때 발생할 수 있는 보안 취약점(로그 노출)이나 파괴적 행위(DB 드랍)를 제어하기 위해 시스템적으로 컨텍스트를 제한하고 있다. 인간의 리뷰에만 의존하지 않고, AI 자체의 작업 컨텍스트에 '가드레일'을 설치하는 것이 최신 트렌드이다.

**➡️ 시사점**
개발팀은 자신들의 프로젝트에서 가장 민감하거나 자주 실수하는 도메인(예: DB 마이그레이션, 결제 로직, 보안 로그)을 식별하고, 해당 디렉터리에서만 발동하는 전용 Rule 파일을 세팅해야 한다.

> "이처럼 각각의 메모리 파일에 조건을 설정함으로써, 특정 작업을 할 때 불필요한 규칙이 아예 클로드 머릿속에 안 들어오게 할 수 있는 거죠." [[04:57](http://www.youtube.com/watch?v=FBv8hK_DtJ8&t=297)]

## [07:55–11:25] 실전 프로젝트 아키텍처 구성 및 시사점

앞서 배운 조건부 자동 로딩 전략을 가상의 '쇼핑몰 프로젝트'에 적용해 보고, 지금 당장 실무자들이 본인의 프로젝트에서 실행해야 할 액션 아이템을 정리한다.

---

### 💡 핵심 내용
대규모 시스템에서는 '항상 로드되는 전역 규칙'과 '특정 상황에만 로드되는 지역 규칙'을 명확히 분리하여, AI가 자신의 현재 업무 매뉴얼에만 온전히 집중할 수 있는 환경을 구축해야 한다.

**📌 근거와 논리**

1. **쇼핑몰 프로젝트 rules 구성 시뮬레이션** —
   모든 규칙을 조건 없이 불러올 경우, 결제 API를 수정하는데 UI 컴포넌트 규칙이 끼어들어 AI의 추론 능력을 낭비하게 된다. 이를 아래와 같이 분리해야 한다.

```text
[쇼핑몰 프로젝트의 이상적인 .claude/rules/ 구성]

.claude/rules/
 ├─ code-style.md          (조건 없음 → 항상 로드됨)
 ├─ react-components.md    (조건: components/** → 화면 파일 작업 시 로드)
 ├─ api-design.md          (조건: api/** → API 파일 작업 시 로드)
 ├─ database.md            (조건: db/** → 데이터베이스 작업 시 로드)
 └─ testing.md             (조건: *.test.ts → 테스트 파일 작업 시 로드)
```

2. **작업별 자동 선택 매커니즘** —
   - **상품 목록 화면 개발 시**: `code-style.md` + `react-components.md` 만 결합되어 클로드에게 전달된다.
   - **결제 API 개발 시**: `code-style.md` + `api-design.md` 만 결합되어 클로드에게 전달된다.
   - 불필요한 테스트 규칙이나 DB 규칙은 아예 배제된다.

3. **수강생의 실제 문제 해결 데이터** — 인프런 클로드 코드 마스터 강의(4개월 만에 4,300명 수강)의 수강 후기에 따르면, 프로 요금제로 매달 4~6만 원을 결제하면서도 늘 토큰 제한(Token Limit)에 걸려 고통받던 수강생들이, 이러한 컨텍스트 윈도우 관리를 도입한 직후 토큰 리밋에 걸리지 않고 AI가 훨씬 똑똑하게 코딩하는 경험을 체감했다고 증언한다.

→ 모든 규칙을 하나의 파일에 병합하면 토큰 소모가 극심해지고 AI의 지시 이행력이 떨어진다 → 도메인별로 Rule을 쪼개고 필요할 때만 호출한다 → 토큰을 획기적으로 절약할 뿐 아니라, 중요한 규칙에 온전히 주의(Attention)를 집중하게 되어 최종 결과물의 품질이 비약적으로 상승한다

**🌐 맥락**
최근 대형 언어 모델(LLM)의 컨텍스트 윈도우가 100K~200K 수준으로 커졌음에도 불구하고, 관련 없는 프롬프트나 노이즈 데이터가 섞여 있으면 모델의 '바늘 찾기(Needle In A Haystack)' 능력이 저하된다. 불필요한 컨텍스트를 스스로 덜어내는 '미니멀리즘 프롬프팅'이 생산성 향상의 핵심 기술로 떠올랐다.
